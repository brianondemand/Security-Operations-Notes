## 🧪 Practical Lab Exercises


### Lab 1: SOC L1 - "A Day in the Life" Simulation

**Objective:** Simulate the daily workflow of a Tier 1 SOC Analyst, from alert triage to remediation planning [](https://github.com/MattSolo7/My-SOC-L1-Training).

**Tools Used:** Simulated SIEM Dashboard (Wazuh), VirusTotal, AbuseIPDB.

**Instructions:**

1. **Access Environment:** Log in to the lab environment and access the simulated SOC dashboard containing security tickets and logs.

2. **Review Security Tickets:** Review the SIEM dashboard for new alerts. Note critical details for each alert: **Alert name, Source IP, Destination IP, Timestamp, and Detection tool** (IDS, Firewall, Email).

3. **Investigate the Alert:** Dig deeper into a specific alert by cross-referencing:
    
    - IDS/IPS logs for signature matches.
    
    - Network traffic data for anomalies.
    
    - Email headers for suspicious attachments or links.
    
4. **Threat Intelligence Lookup:** Take any suspicious IP addresses or file hashes and look them up in **VirusTotal** and **AbuseIPDB** to verify if they are known IOCs [SOC Lab](https://github.com/MattSolo7/My-SOC-L1-Training).

5. **Plan Remediation:** Based on your findings, outline a containment and remediation strategy (e.g., block the IP, isolate the host, reset credentials).

### Lab 2: SIEM Mastery with Splunk

**Objective:** Build a home SOC lab, ingest security data, and create detections for common attacks using Splunk [SOC Lab](https://github.com/r-ramos2/SOC-Lab-at-Home-with-Splunk-Comprehensive-SIEM-Threat-Detection-and-Response-Guide). [Splunk Guide](https://github.com/r-ramos2/SOC-Lab-at-Home-with-Splunk-Comprehensive-SIEM-Threat-Detection-and-Response-Guide/blob/main/README.md).

**Tools Used:** Splunk Enterprise, BOTS v3 Dataset.

**Part A: Setup and Data Ingestion**

1. **Install Splunk Enterprise** on a local machine or VM.

2. **Create an Index:** In Splunk, navigate to `Settings` > `Indexes` and create a new index named `soc_lab` [SOC Lab](https://github.com/r-ramos2/SOC-Lab-at-Home-with-Splunk-Comprehensive-SIEM-Threat-Detection-and-Response-Guide).

3. **Ingest Data:** Download the BOTS v3 dataset. Use the Splunk interface to upload this data, directing it to your `soc_lab` index.


**Part B: Threat Detection**

1. **Brute Force Detection:** Write an SPL query to detect potential brute-force attacks by finding source IPs with more than 100 failed login attempts.

```spl
index="soc_lab" action=login status=failed
| stats count by src_ip
| where count > 100
``` [citation:3]
```


2. **Lateral Movement:** Identify potential lateral movement by finding internal systems (`dest_ip=10.0.0.0/8`) that have an unusually high number of connections from another internal IP.

```spl
index="soc_lab" dest_ip=10.0.0.0/8
| stats count by src_ip, dest_ip
| where count > 50
``` [citation:3]
```


**Part C: Automation and Visualization**

1. **Create an Alert:** Configure an alert for the brute force search above. Set it to trigger every hour and have it send a test email alert.

2. **Build a Dashboard:** Create a new dashboard with a panel showing the "Top 10 Failed Login IPs" using a bar chart [Threat Detection Guide](https://github.com/r-ramos2/SOC-Lab-at-Home-with-Splunk-Comprehensive-SIEM-Threat-Detection-and-Response-Guide). [SOC Lab](https://github.com/r-ramos2/SOC-Lab-at-Home-with-Splunk-Comprehensive-SIEM-Threat-Detection-and-Response-Guide/blob/main/README.md).

### Lab 3: Network Defense with Snort

**Objective:** Deploy and configure the Snort IDS, write custom rules, and test them against simulated attacks [AH Protection](https://github.com/kinnaridobariya/Advanced-Hacking-Prevention)
[Snort Rules](https://www.pluralsight.com:443/labs/aws/custom-snort-rules).

**Tools Used:** pfSense, Snort, Attacker/Victim VMs.  
**Estimated Time:** 2 hours

**Part A: Setup**

1. **Deploy Environment:** Set up two virtual machines on the same network (e.g., Ubuntu for attacker, Metasploitable for victim).

2. **Install Snort:** Install Snort on the victim machine or a separate monitoring VM.


**Part B: Writing Custom Rules**

1. **ICMP Scan Detection:** Write a Snort rule to alert on any ICMP traffic (ping sweeps) from the attacker to the victim network. Save the rule in your local rules file.

```
alert icmp $EXTERNAL_NET any -> $HOME_NET any (msg:"ICMP Ping Detected"; sid:1000001; rev:1;)
``` [citation:9]
```

2. **HTTP GET Suspicious Request:** Write a rule to detect suspicious HTTP GET requests containing "cmd=", indicative of a web shell attempt.

```
alert tcp $EXTERNAL_NET any -> $HTTP_SERVERS $HTTP_PORTS (msg:"Suspicious HTTP GET"; content:"GET"; http_method; content:"cmd="; http_uri; sid:1000002; rev:1;)
``` [citation:9]
```


**Part C: Testing and Tuning**

1. **Generate Traffic:** From the attacker machine, run a simple ping sweep (`ping -c 4 [victim_ip]`) and use `curl` to send a malicious-looking HTTP request (`curl "http://[victim_ip]/index.php?cmd=whoami"`).

2. **Analyze Alerts:** Check the Snort alert logs (`/var/log/snort/alert`). You should see alerts for your new rules.

3. **Reduce False Positives:** If your HTTP rule is too noisy, tune it by adding the `flow:established,to_server` keyword to only alert on established connections, reducing alerts from scans or random noise [Custom Snort Rules](https://www.pluralsight.com:443/labs/aws/custom-snort-rules).

### Lab 4: Strategic Incident Response - Tabletop Exercise

**Objective:** Simulate a cross-departmental response to a major security incident to test communication, decision-making, and playbook effectiveness [](https://www.csoonline.com/article/3839258/suite-404-training-the-cyber-gau-in-a-fun-way.html)[](https://hitachicyber.com/case-study/strengthening-cyber-resilience-through-tabletop-exercises-for-an-apac-it-tech-services-company/#awb-open-oc__109).

**Tools Used:** Scenario cards, whiteboard/paper, incident response playbook.  
**Estimated Time:** 1 hour (for the exercise) + 30 min debrief

**Scenario:** You are the executive team of a luxury hotel group. Your IT systems have been acting strangely [Training](https://www.csoonline.com/article/3839258/suite-404-training-the-cyber-gau-in-a-fun-way.html).

**Instructions:**

**Phase 1: Triage**  
The front desk reports that the electronic key card system for guest rooms is intermittently failing. Your team must assess the impact on **Service, Reputation, Sales, and Cybersecurity**. Do you treat this as a major incident?

**Phase 2: Investigation**  
Your IT team pulls logs and finds three anomalies:

1. An unauthorized admin login from an overseas IP at 3:00 AM.
    
2. A large, encrypted outbound data transfer to a cloud storage site the company doesn't use.
    
3. A scheduled task was created on the domain controller overnight.[](https://www.csoonline.com/article/3839258/suite-404-training-the-cyber-gau-in-a-fun-way.html)  
    Your task: Based on these logs, determine the scope and nature of the breach.
    

**Phase 3: Response**  
Suddenly, a hacktivist group claims responsibility and defaces your website. The CEO demands action. You are presented with two choices for each new incident. You must choose one.

- **Incident:** A major influencer posts about the hotel's Wi-Fi being down on TikTok, and the BBC calls for a statement.
    
    - **Option A:** Pause all technical response to prepare a PR statement.
        
    - **Option B:** Delegate PR to a comms lead and continue focusing on technical containment.[](https://www.csoonline.com/article/3839258/suite-404-training-the-cyber-gau-in-a-fun-way.html)
        
- **Debrief:** After 30 minutes of play, stop and review the decisions. Discuss what went well and what the team would do differently next time [](https://hitachicyber.com/case-study/strengthening-cyber-resilience-through-tabletop-exercises-for-an-apac-it-tech-services-company/#awb-open-oc__109).
