# Windows Logging

Logs are records of events that occur on your computer, whether initiated by a person or a running process. They assist in tracking activities and troubleshooting issues. The Windows event log includes entries from the operating system and applications like SQL Server or Internet Information Services (IIS). These logs are in a structured data format, making them easy to search and analyze. Additionally, some applications, such as IIS, generate log files in text format, like IIS Access Logs.

This article examines the Event Viewer interface and its features, and introduces other major application and service logs. Examples are provided to help you understand how monitoring events can aid in managing your systems’ health and security.

### Windows Event Logs

**Windows Event Viewer** allows you to view and manage Windows event logs. With this application, you can browse through the logs, search for specific entries, filter different log types, and export logs for further analysis. We’ll guide you on how to access Windows Event Viewer and explore its various features.

### Starting Windows Event Viewer

Windows Server 2019 Event Viewer can be accessed in several ways:

- Windows Control Panel
- Server Manager
- Windows Admin Center
- Computer Management
- Component Services
- Command Prompt

### Windows Control Panel

Control Panel is a standard Windows feature used for viewing and modifying system settings. It is available in both Windows Server and Windows desktop editions. To access the Event Viewer:

1. Open **Control Panel**
2. Click **Administrative Tools**
3. Double-click **Event Viewer**

The Server Manager console lets you manage settings on the local server and on remote servers. To access Event Viewer from Server Manager:

### Using the Windows Event Viewer Interface

Event Viewer has an intuitive user interface. The main screen is divided into three sections:

- Navigation pane
- Detail pane
- Action pane

You can create Summary and Custom views. We’ll guide you through these options.

### Navigation Pane

The Navigation pane is where you choose the event log to view. By default, there are five categories of Windows logs:

- ***Application*** — Information logged by applications hosted on the local machine.
- ***Security*** — Information related to login attempts (success and failure), elevated privileges, and other audited events.
- ***Setup*** — Messages generated when installing and upgrading the Windows operating system. If the Windows system is a domain controller, those messages are also logged here.
- ***System*** — Messages generated by the Windows operating system.
- ***Forwarded Events*** — Events forwarded by other computers when the local machine is functioning as a central subscriber.

There is also a section for Applications and Services Logs, including categories for Hardware Events, Internet Explorer and Windows PowerShell events.

To manage a large log, you can use the ***Clear Log*** ***action*** to delete all stored events. ### Exporting Events

You can click “ ***Save All Events As*** ” or “ ***Save All Events in Custom View As*** ” to export events from the current log to an event file with an EVTX extension. This functionality is useful in several scenarios: for instance, if you need to send your system’s health status to a third-party vendor, you can provide them with an exported event file.

Alternatively, you can archive your logs before deleting them or send your saved logs to a centralized backup medium. Saving event logs to an event file is a convenient feature. Administrators can open a saved log by clicking “ ***Open Saved Log*** ” and navigating to the log’s location.

### Failover Cluster Manager

The ***Windows Server Failover Clustering*** service allows multiple Windows servers to operate together as a cluster, ensuring fault tolerance. If one server experiences a hardware failure, the service detects it and seamlessly transitions operations to another server. This process includes automatically redirecting all network traffic to the functional server, maintaining a highly available environment. In a cluster, applications connect through a virtual IP or cluster name, with Windows managing traffic routing to the appropriate server. In the event of a failure, applications continue to run smoothly. This clustering service is essential for modern SQL Server high-availability solutions, such as ***AlwaysOn Availability Groups***.

The Failover Cluster Manager, an integrated Windows tool, comes equipped with its own Event Viewer. With this Event Viewer, system administrators can diagnose issues when their cluster experiences failures or malfunctions. In the Navigation pane, selecting the Cluster Manager event viewer node reveals cluster-related events, as depicted in the accompanying screenshot.

### IIS Access Logs

The *access logs of Internet Information Services* contain details about requested URIs and the status indicating the success of the response. These logs are saved as files in the W3C Extended Log Format, which is a form of comma-separated values (CSV). The location of the log files is determined by the settings in the IIS Manager Logging section. By default, the location is:

```c
%SystemDrive%\inetpub\logs\LogFiles
```

For example, here’s a log file on **C:**, with **W3SVC1** as the virtual host and **u\_ex150428** as a file name coded with the date 2015–04–28:

```c
C:\inetpub\logs\LogFiles\W3SVC1u_ex150428.log
```

Here’s an excerpt from the log file. The column definition is in a comment. The request for **/manager/html** returned a **404** status code as the page doesn’t exist.

```c
#Software: Microsoft Internet Information Services 7.5

#Version: 1.0

#Date: 2015-04-28 12:12:05

#Fields: date time s-ip cs-method cs-uri-stem cs-uri-query s-port cs-username c-ip

2015-04-28 02:51:38 10.211.14.109 GET /manager/html - 80 - 222.186.56.21 Mozilla/5.0+(compatible;+MSIE+10.0;+Windows+NT+6.2;+WOW64;+Trident/6.0) - 404 0 64 4850
```

### Using PowerShell with Windows Logs

Windows PowerShell offers administrators a scriptable method for engaging with Windows logs. Instead of the outdated Get-EventLog, which relies on a deprecated Win32 API, we’ll utilize Get-WinEvent in our demonstrations. Through Get-WinEvent, one can access diverse Windows logs such as Application, System, and Security logs.

### Listing available logs

To list all available logs on a system, use this command:

`Get-WinEvent -ListLog *`

The output will look similar to this:

![](https://miro.medium.com/v2/resize:fit:640/format:webp/0*N2VfBuM8RulX5r1i.png)

Here is what each of the column names means:

- LogMode — How logs are (or aren’t) overwritten when the log is full. Circular means a first-in first-out (FIFO) style where new logs automatically overwrite old logs/
- MaximumSizeInBytes — The maximum size of the log file.
- RecordCount — The number of individual records in the log file.
- LogName — The name of the logfile

### Viewing Event Logs

To list logs on a Windows system, you can use the command:

`Get-WinEvent -LogName <LogName>`

Where `<LogName>` is the name is one of the logs in the output of Get-WinEvent -ListLog \*.

For example, to view all the Application log records on a local Windows system, use this command:

`Get-WinEvent -LogName 'Application'`

In many cases, log files contain hundreds of lines of records. To limit how many records are displayed, use the -MaxEvents parameter. For example, to view only the last 5 Application log records, use this command:

`Get-WinEvent -LogName 'Application' -MaxEvents 5`

![](https://miro.medium.com/v2/resize:fit:640/format:webp/0*Y9ew1_ANFDhPlTed.png)

### Filter for specific Event IDs

If you know what log file and event ID you’re looking for, [PowerShell hashtables](https://learn.microsoft.com/en-us/powershell/scripting/learn/deep-dives/everything-about-hashtable?view=powershell-7.2) make it easy to filter through Windows Logs. Hashtables use key/value pairs to simplify PowerShell queries.

For example, to list all Security log entries with ID 4672 (Special privileges assigned to new logon), use this command:

`Get-WinEvent -FilterHashtable @{ Logname='Security'; Id='4672' }`

![](https://miro.medium.com/v2/resize:fit:640/format:webp/0*rMv3etnmemCkPgWf.png)

### Filter based on date and time

To further refine your PowerShell log searches, you can use the [Get-Date cmdlet](https://learn.microsoft.com/en-us/powershell/module/microsoft.powershell.utility/get-date?view=powershell-7.2) in your query. For example, to limit our query of Security log entries with ID 4672 to the last hour, we can use this command:

`Get-WinEvent -FilterHashtable @{ Logname='Security'; Id='4672'; StartTime=(Get-Date).AddHours(-1) }`

![](https://miro.medium.com/v2/resize:fit:640/format:webp/0*vj7WtniDiCKVZCMq.png)

The Get-Date cmdlet allows for similar filters for other units of time as well. They are:

- (Get-Date).AddTicks
- (Get-Date).AddMilliseconds
- (Get-Date).AddSeconds
- (Get-Date).AddMinutes
- (Get-Date).AddDays
- (Get-Date).AddMonths
- (Get-Date).AddYears
